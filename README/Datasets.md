# Datasets

This document describes how datasets are handled in this project and how to add new datasets for Gemma multimodal fine-tuning.

## Dataset Structure

**Path resolution:** `data/datasets/<dataset_name>/` and the patch base `data_patches/` (overrides, blacklists) use the **same rule**: resolved relative to the **repository root** when no `config.ini` exists in the process current working directory (for example when you run `gemma-macos-tuner finetune` with `--config` pointing at the project `config.ini` from another directory). If `config.ini` is present in cwd, paths are relative to cwd instead (local workflows and tests). Avoid a stray unrelated `config.ini` in cwd if your data and patches live only under the repo — it would switch resolution to cwd-relative.

The general structure for datasets is as follows:

```
data/
    datasets/
        {dataset_name}/
            {dataset_name}.csv         # Original, unprocessed CSV data
            {dataset_name}_prepared.csv # Processed CSV with local audio paths
    audio/
        {dataset_name}/                # (Optional) if you want to separate by dataset
            {note_id}.wav              # Decoded audio files
        {note_id}.wav                  # Decoded audio files (if not using per-dataset audio folders)
    splits/
        {dataset_name}/
            train.csv                  # Training split (note_ids)
            validation.csv             # Validation split (note_ids)
```

*   **`data/datasets/{dataset_name}/`:** Contains the dataset-specific files.
    *   **`{dataset_name}.csv`:** The original, unprocessed CSV file containing at least the following columns:
        *   `note_id`: A unique identifier for each audio sample.
        *   `audio_url`: The URL to the original audio file (typically `.m4a`).
        *   `text_verbatim`: The verbatim transcription (including disfluencies).
        *   `text_perfect`: The cleaned transcription (excluding disfluencies).
        *   `speaker_id` (optional): An identifier for the speaker.
    *   **`{dataset_name}_prepared.csv`:** A processed version of the CSV file, generated by `prepare_data.py`. It contains the following columns, **with** a header row:
        1. `id`
        2. `audio_path` (local relative path to the decoded `.wav` file, NOTE: always use '/' regardless of platform)
        3. `text_verbatim`
        4. `text_perfect`
        5. `speaker_id` (if available in the original data)
        6. `language` (ISO 639-1 language code or '??' for unknown)
        7. `duration` (actual calculated audio duration in seconds)

*   **`data/audio/`:** Contains the downloaded and decoded audio files.
    *   Audio files are named `{note_id}.wav`.
    *   You can optionally create subdirectories for each dataset under `data/audio/` (e.g., `data/audio/data3/`) if you prefer to keep audio files separated by dataset.

*   **`data/splits/{dataset_name}/`:** Contains CSV files defining the training and validation splits.
    *   **`train.csv`:** Contains a list of `id` for the training set, along with all other columns from the prepared CSV.
    *   **`validation.csv`:** Contains a list of `id` for the validation set, along with all other columns from the prepared CSV.

## Data Preparation

The `scripts/prepare_data.py` script handles the data preparation process, including:

1. **Downloading Audio:** Downloads the audio files from the URLs in the `{dataset_name}.csv` file.
2. **Decoding Audio:** Decodes the downloaded audio files (typically `.m4a`) to `.wav` format using `ffmpeg`.
3. **Creating Prepared CSV:** Generates the `{dataset_name}_prepared.csv` file with local audio paths and the correct column format.
4. **Generating Splits:** Creates `train.csv` and `validation.csv` split files based on a random split or a speaker-based split (if `speaker_id` is available and configured in `config.ini`).
5. **Duration Calculation and Filtering:** Calculates the actual duration of each audio file using `librosa` and filters out samples exceeding `max_duration` specified in `config.ini`. The calculated duration is stored in the `duration` column of the prepared CSV.
6. **Data Validation:** Performs checks for missing values, audio file integrity, and other potential issues.
7. **Language Filtering:** Filters the dataset based on the `languages` setting in `config.ini`. See the "Language Handling" section below for details.
8. **Applying Overrides and Blacklists:** Applies overrides and blacklists from CSV files located in `data_patches/{dataset_name}/`. See the root [`README.md`](../README.md) and dataset utilities in `gemma_tuner/utils/dataset_utils.py` for details.

**To prepare a dataset:**

1. **Place the raw CSV file:** Put your raw CSV file (e.g., `data123.csv`) in the `data/datasets/data123/` directory.
2. **Configure `config.ini`:**
    *   Add a `[dataset:{dataset_name}]` section to your `config.ini` file if the dataset needs specific settings.
    *   Update `[dataset_defaults]` with appropriate values.
    *   If you want to keep audio files separated by dataset, create a subdirectory `data/audio/data123/` and update `audio_dir` accordingly.
    *   Configure the `languages` and `language_mode` settings as described in the "Language Handling" section.
3. **Run `prepare_data.py`:** Execute `gemma-macos-tuner prepare data123 --config config.ini` to prepare the new dataset.
4. **Update `gemma_tuner/models/gemma/finetune.py` (if needed):** If your new dataset requires special handling during finetuning (e.g., different preprocessing steps), extend the Gemma training path there or in `gemma_tuner/utils/gemma_dataset_prep.py`.

## Language Handling

The framework supports flexible language handling during both data preparation and finetuning. The behavior is controlled by the `language_mode` and `languages` settings in the `[dataset:...]` or `[profile:...]` sections of `config.ini`.

### `language_mode`

This setting determines how the language is handled during finetuning:

*   **`mixed`:** No language token is set during finetuning. The model is expected to handle mixed-language input. This is suitable for datasets where the language is unknown or can change within a single audio sample.
*   **`strict`:** The language token is set based on the `language` field of each sample in the dataset. This assumes that the `language` field is accurate and reliable.
*   **`override:XX`:** The language token is forced to `XX` (where `XX` is a two-letter language code or `??`) for all samples, regardless of the `language` field in the dataset. This is useful for finetuning on a specific language or when the language information in the dataset is unreliable.

### `languages`

This setting determines which languages are included in the prepared dataset during the `prepare_data.py` stage.

*   It can be set to `all` to include all languages.
*   It can be a comma-separated list of two-letter language codes (e.g., `en,es,pt`).
*   It can include the special token `??` to include samples where the language is unknown or not specified.

**Filtering Logic:**

*   During data preparation, samples are filtered based on the `languages` setting.
*   If `languages` is set to `all`, no filtering is performed.
*   If `languages` is a list of language codes, only samples with a `language` field that matches one of the specified codes **or** is equal to `??` (if `??` is in the list) are included.
*   Samples with languages not in the `languages` list and not equal to `??` are **excluded**.

**Example Scenarios:**

1. **`languages = en,es` and `language_mode = mixed`:** The prepared dataset will only include samples with `language` equal to `en`, `es`, or `??`. During finetuning, no language token will be set, and the model will be trained on a mix of English, Spanish, and unknown-language samples.
2. **`languages = en,es` and `language_mode = strict`:** The prepared dataset will be the same as in scenario 1. During finetuning, the language token will be set based on the `language` field of each sample (either `en` or `es`).
3. **`languages = en,es,??` and `language_mode = strict`:** The prepared dataset will include samples with `language` equal to `en`, `es`, or `??`. During finetuning, the language token will be set based on the `language` field, with `??` likely being treated as a special "unknown language" token.
4. **`languages = all` and `language_mode = override:es`:** The prepared dataset will include all samples regardless of their language. During finetuning, the language token will be forced to `es` for all samples.
5. **`languages = en` and `language_mode = override:??`:** The prepared dataset will include only english samples. During finetuning, the language token will be forced to `??` for all samples.

**Note:** The `??` token is used to represent unknown or unspecified languages. It's treated as a special language code during filtering and can be used in `language_mode = strict` or `override:??` to handle cases where the language is not known.

## Adding New Datasets

To add a new dataset (e.g., `data123`):

1. **Create Dataset Directory:** Create a new directory under `data/datasets/` for your dataset (e.g., `data/datasets/data123/`).
2. **Place Raw Data:** Place your raw CSV file (e.g., `data123.csv`) in the dataset directory. The CSV file should have at least the following columns: `note_id`, `audio_url`, `text_verbatim`, and `text_perfect`.
3. **Update `config.ini`:**
    *   Add a new dataset configuration section (e.g., `[dataset:data123]`) to your `config.ini` file if the dataset needs specific settings.
    *   Update `[dataset_defaults]` with appropriate values.
    *   If you want to keep audio files separated by dataset, create a subdirectory `data/audio/data123/` and update `audio_dir` accordingly.
    *   Configure the `languages` and `language_mode` settings as described in the "Language Handling" section.
4. **Run `prepare_data.py`:** Execute `gemma-macos-tuner prepare data123 --config config.ini` to prepare the new dataset.
5. **Update `gemma_tuner/models/gemma/finetune.py` (if needed):** If your new dataset requires special handling during finetuning (e.g., different preprocessing steps), extend the Gemma training path there or in `gemma_tuner/utils/gemma_dataset_prep.py`.

## Profile keys: modality, text, and image fine-tuning

These keys can appear in `[DEFAULT]`, `[dataset:…]`, `[model:…]`, or `[profile:…]` (profile wins last). They default so that **existing audio profiles behave unchanged** without edits.

| Key | Default | Meaning |
| --- | --- | --- |
| `modality` | `audio` | `audio` — speech + transcript (current behavior). `text` — text-only CSV fine-tuning (instruction or completion). `image` — image + text CSV (captioning or VQA); see image columns below. `audiovisual` — joint audio + image + text in one sample (caption-style only); requires `audio_path` and the column named by `image_path_column` to be present in the CSV. |
| `text_sub_mode` | `instruction` | When `modality = text`: `instruction` — prompt + response columns with prompt masked from loss; `completion` — single text column, full sequence trained. |
| `prompt_column` | *(empty / none)* | When `text_sub_mode = instruction`, the column name for the user/prompt text. Ignored for `completion`. When `modality = image` and `image_sub_mode = vqa`, the column name for the **question** text. |
| `max_seq_length` | `2048` | Maximum token length for text-mode batches (padding/truncation). |
| `image_sub_mode` | `caption` | When `modality = image`: `caption` — fixed instruction + caption/response in `text_column`; `vqa` — question in `prompt_column`, answer in `text_column`. |
| `image_path_column` | `image_path` | CSV column for the image file path (local or supported URI). |
| `image_token_budget` | `280` | Vision token budget for the processor. Must be one of `70`, `140`, `280`, `560`, `1120` (train/serve must match). |

### `modality = audiovisual` (joint image + audio + text)

Single training run over rows that have *both* an `audio_path` and an image file (`image_path_column`, default `image_path`), with a free-form caption in `text_column`. The collator builds a fixed user prompt — "Describe what is happening using the attached audio and image." — and supervises on the row's caption.

Constraints:

- Caption-style only. `image_sub_mode = vqa` is rejected at config time for this modality.
- Local CSV datasets only (same gate as `modality = image`).
- `image_token_budget` applies as in image mode.

Example profile:

```ini
[profile:my-audiovisual-lora]
model = gemma-4-e2b-it
dataset = my-av-dataset
modality = audiovisual
image_path_column = image_path
text_column = label
per_device_train_batch_size = 4
```

See `config/config.ini.example` for commented examples.

## Dataset Loading

The `utils/dataset_utils.py` script contains the `load_dataset_split` function, which is responsible for loading a dataset into a Hugging Face `datasets.Dataset` object. It takes the following arguments:

*   **`split`:** The dataset split to load (`train` or `validation`).
*   **`dataset_config`:** The dataset configuration dictionary from `config.ini`. The `load_dataset_split` function takes the dataset name from this dictionary to locate the correct prepared CSV and split files under `data/datasets/{dataset_name}/`.
*   **`max_train_samples`:** (Optional) Limits the number of samples loaded from the specified split.

The function directly loads the specified split file (`train.csv` or `validation.csv`) using `datasets.load_dataset("csv", data_files={split: split_path})`.

**Overrides and Blacklists:**

The `load_dataset_split` function automatically applies overrides and blacklists defined in the `data_patches/{dataset_name}` directory. See the root [`README.md`](../README.md) for patch layout and `gemma_tuner/utils/dataset_utils.py` for behavior.

## Dataset Evaluation

You can evaluate a model on a specific dataset using the `gemma-macos-tuner evaluate` command. There are two ways to specify the model and dataset:

1. **Using a profile:**

    ```bash
    gemma-macos-tuner evaluate medium-data3
    ```

    This will evaluate the finetuned model specified in the `medium-data3` profile on the dataset defined in that profile.

2. **Using a model+dataset combination:**

    ```bash
    gemma-macos-tuner evaluate gemma-4-e2b-it+data3
    ```

    This will evaluate using the `gemma-4-e2b-it` model entry on the `data3` dataset. The evaluation will use the settings defined in the `[model:gemma-4-e2b-it]` and `[dataset:data3]` sections of `config.ini`. The results will be saved under `output/` for that run profile.

    **Note:** Only datasets defined in `config.ini` can be used in this mode.

## Notes

*   The `max_train_samples` argument can be used with `gemma-macos-tuner finetune` or `gemma-macos-tuner evaluate` to limit the number of samples loaded for either training or evaluation.
*   The `speaker_id` column is optional. If it's not present in the original data, the scripts will still work, but speaker-based splitting will not be available.
